1: Outline

RTA1 emulated machine is available for two POSIX platforms
all OSX PowerPC and macOS Intel platforms

any Intel PC on which you have installed openSUSE Linux or another Linux
A network interface process and an emulator process constitute the RTA1 emulated machine

RTA1 machine executes a target system image with webserver, snmp agent, telnet server,
and embedded UDP servers for 96-bit and 192-bit floating arithmetic.
Client-side UDP floating arithmetic applications are packaged for OSX, Linux and Windows

	$HOME/Downloads/rta

2.2: Required Utilities and Path Setup

RTA1 emulators are in rta/binary.smp

RTA1 utilities are in rta/binary.rta and are frequently required

Subdirectories in emulators and utilities paths are [ suse.x64 suse.x32 osx.x64 osx.x32 osx.ppc ]

Platform depending rta1 operation requires paths in $PATH for RTA1 emulators and utilities

Environmental names [ RTA_BINARY RTA_BIN_64 RTA_BIN_SMP ] are referenced in some shell scripts and optionally in $PATH

RTA1 uses many just-in-time assemblies and does not run without separate product masmx7r3A

RTA1 requires masmx7r3 Build 12 or later

masmx7r3A Build 12 and Build 12-64 currently download from http://timmilescox.github.io/masmx/

masmx/masmx.7r3/masmx.7r3/hosts contains binaries in [ suse.x64 suse.x86 osx.x64 osx.x86 osx.ppc ]

Whichever runs on the platform is required in $PATH and sometimes its path has the variable name $MASMX

Components and utilities of RTA1 emulator and masmx are present for

  • 32-Bit Intel OSX

  • 32-Bit PowerPC OSX
  • 32-Bit Intel Linux
  • 64-Bit Intel Linux

2.3: SMP Emulators by Platform

RTA1 emulators Stepping 01.01.2024 [ USA 01.01.2024 ] are newly constructed for higher performance and Symmetric Multicore

New SMP emulators use available core threads of the emulator platform and are supplied in variants of [ 1 2 4 8 ] cores

Supplied emulators are mostly for Intel 64-bit and 32-bit platforms running Linux or macOS

The only RTA1 emulator supplied for macOS PowerPC is a completely new SMP * 1-core emulator. Project RTA1 will release SMP emulator variants [ 2 4 8 ] for PowerPC subject to resourcing machine-code upgrades

Four RTA1 emulators for 64-bit Intel macOS RTA1 are supplied. Environment variable $RTA_SMP = [ 1 2 4 8 ] selects [ rta1_1 rta1_2 rta1_4 rta1_8 ] for execution

These and macOS 32-bit PowerPC [ rta1_1 ] acquire storage per $RTA_MBANKS (262144-word blocks of RTA1 executable space) at startup

There is a difference between macOS releases whether sudo command in runscript ./r needs -E option to transfer environment variables to the emulator. In macOS for PowerPC this option may be unknown and not needed

RTA emulators for Linux 64-bit Intel and for Linux and macOS 32-bit Intel are supplied in multiple cores * storage size variants and selected at runtime. Emulator is selected for execution using variables $RTA_SMP and $RTA_MBANKS

Depending on platform available emulators are 


	$RTA_BINARY/rta1_[ 1 2 4 8 ].[ 1 2 4 8 32 64 256 1024 2048 ]
	32-bit Intel macOS and Linux	run as script rta/tgen.smp/ra	[-[R][Z][s]] rta1_1|2|4|8.2048 are macOS only

	$RTA_BIN_64/rta1_[ 1 2 4 8 ].[ 1 2 4 8 32 64 256 1024 2048 4096 8192 ]
	64-bit Intel Linux		run as script rta/tgen.smp/xra	[-[R][Z][s]]

	$RTA_BIN_64/rta1_[ 1 2 4 8 ]
	64-bit Intel macOS		run as script rta/tgen.smp/r	[-[R][Z][s]]

	$RTA_BINARY/rta1_1
	32-bit PowerPC OSX		run as script rta/tgen.smp/rap	[-[R][Z][s]]

64-bit Intel macOS emulators can be run with RTA_MBANKS=65536 (RTA1 executable space 16 Gigawords) but swap requirements may make demands on platform volume

64-bit Intel macOS emulators and 32-bit PowerPC OSX emulator have variants [ rta1_1A rta1_2A rta1_4A rta1_8A ] which include the touchpoint trace feature (a memory-write equivalent of breakpoint). touchpoint feature is included in all 32-bit Intel emulators and Linux 64-bit Intel emulators. [ rta1_1A rta1_2A rta1_4A rta1_8A ] contain less Intel-specific source code than [ rta1_1 rta1_2 rta1_4 rta1_8 ] and are the code base of emulations for platforms not similar to Intel

3: Network Topology

Script .rta_net must be copied from { linux.net | linuxnet.x64 | portal | portal.x64 } to $HOME

Adjust values in $HOME/.rta_net RTA_ENET and RTA_WIFI if you need different addresses from 172.29.7.8 and 172.29.7.66

This becomes necessary if more than one RTA1 is on the local net

Supplied routes have addresses 172.29.7.8 .. 172.29.7.62 in wired Ethernet and addresses 172.29.7.66 .. 172.31.255.254 in WiFi

These routes may need to be changed in $HOME/.rta_net for traffic from RTA1 and for traffic from platforms to RTA1 in rta/client/rundum

5: Routing to RTA1

Appendix B: Autoconf and Multihomed Routes

routes in the route table of RTA1 target image are typed homed (bound to an interface) or not homed

routes not homed may replicate remote network addresses in separate attached networks

routing picks a next hop for datagrams, and if the route is homed IP uses the indicated outgoing interface

if the route is not homed the outgoing interface is chosen according to datagram source network address, which should not be duplicated in any way on one host

RTA1 IPv4 can consequently exchange datagrams with remote sockets which MAY replicate addresses on separate attached networks

ZeroConf is a procedure which may negotiate identical addresses within separate attached networks, but is not the only method of so doing. Manual configuration and DHCP might do the same so long as the networks remain separate and are directly attached

Addresses which potentially replicate in different attached networks are not routable. Routes which could reach replicated addresses are known to RTA1 IPv4 because their interface attribute in the routing table has the wildcard value ANYNET

This allows a single routing table entry to route to an autoconf address however many attached networks replicate that address

Routes of this kind only apply to locally attached networks and are always direct. The next-hop is the the destination

ARP caches each protocol address separately for each attached network where it appears

ARP resolution requires a hit on protocol address plus interface index

Appendix D: readout Utility and Hot Debug

D.1: General Application of readout Utility

readout utility allows selective walk and display of RTA1 system memory pages of 4096 or 262144 words

readout application oes not display NVRAM, network trunks or filestore arrays

Data files (and remote files) can be browsed in dump format with embedded view application

Network trunks and RTA1-local filestores can be examined structurally with debug console in RTA1 machine single step, which is not networked. Debug is the platform startup shell in rta/tgen.smp

I / O Port 124 ram$boundary contains the lowest RAM Page Index which is also the lower limit of readout in Device Zero (executable space)

NVRAM in the pages lower than ram$boundary may contain information which should not be scanned online

Peripheral arrays can be accessed if they are of system memory type but an attempt to access a network trunk array or a filestore array will automatically halt readout. readout can then be run again to diagnose the halt. The cause is that only ISRs can base filestore arrays and network trunks in address windows. Applications can freely call ISRs which do that, but readout executes online and as a matter of policy network users may not do raw scans of filestore devices. readout does not check device types but allows itself to be crashed on illegal access
