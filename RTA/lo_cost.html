1: What are the Machine-Code Emulators?

RTA1 is a project aimed at low-power semiconductor, a freeware programmable processor architecture introduced in emulation on PCs

Emulators run on personal computers for evaluation and RTA1 target software development

RTA1 emulators are now SMP using core threads available on the platform. Nearly all are in Intel machine code.
Thanks to Intel's icc compiler the remaining emulators derived from the C code-base are also rapid and cool-running

These new efficient emulations are closer to the RTA1 project's performance objectives for future semiconductor RTA1s

And certainly more suitable for a project about low power

The new emulators are twice as fast and the platforms work half as much

Newer emulators execute RTA1 instructions in a tight loop of platform machine-code

The result is double the performance of earlier RTA1 emulators and half the demand on platforms

These faster and cooler emulators are initially delivered for Intel and macOS 64-bit and 32-Bit platforms

RTA1 project isn't about emulators, but efficiency is suitable for a project about performance from low power

The density and efficiency of RTA1 target instruction code is extraordinary. Few instructions are issued to get work done

TCP / UDP / IP are implemented in 4000 instructions

Instructions are simple yet RTA1 has the largest floating-point number capability in the world

Floating point in RTA1 emulator does not mean the floating capability of the emulator platform

RTA1 has giant numeric capability unknown to workstations or even to prestige national computers

Emulated implementations run networked applications in continuous server mode and only ever stop for software upgrades or faster emulators in years of running

The free download includes emulated machine, network interfacing, target software, and PC-based development and client softare

The aim of RTA1 is large capability for very little energy. Like when you leave stuff running off a watch battery for a year or more

2: Where are the New Emulators and how do you have Them?

New fast emulators for 64-bit Intel Linux are run with script tgen.smp/xra

New fast emulators for 64-bit macOS Intel are run with script tgen.smp/r

New fast emulators for 32-bit Intel for Linux and macOS are run with script tgen.smp/ra

One shell has script runL executing in directory { rta/{ linux.net | linuxnet.x64 | portal[.x64] }. That trunks network traffic between RTA1 and the world

Another shell for the emulated machine should be in directory rta/tgen.smp

Script tgen.smp/{ra|xra|r} starts an emulator in platform machine code

You need a third shell in rta/client to start routing to RTA1 with script ./prun

That routes to the RTA1 you just launched and any more RTA1s in your network

For more detail see Starting the Emulator

3: RTA1 Emulators Delivered by Platform

3.1: Things you need

Note that RTA1 and fponline do just in time assemblies and won't run without masmx 7r3A

3.2: 32-Bit and Clock-Wraparound-Year

Note that 32-bit emulators have a fix for future clock-wraparound, but this will only work if the clock is allowed to wrap around

Some platforms police the clock to jam at the start of wraparound-year

3.3: Delivered Emulators

Note 64-bit Linux platforms may need a 32-bit runtime download 

	The numbers on the end of emulator names are
	how many 262144-word blocks of excutable space an emulator has

	bash loads the emulator of size corresponding to $RTA_MBANKS

	all emulators pass the same benchmarks

	the largest emulators use a lot of PC virtual space, but don't use very much of it at once

	For each platform the network service process is mentioned first
	as a reminder that it must start before the emulator.

	If the emulator is restarted -R option at the emulator

		 ./ra -R

	will message the network service to resynchronise the trunk

	The emulators, network interfacers, PC and client programs are


	Linux 32-bit
	____________


        network service
        linuxnet                                start in rta/linux.net  ./runL	[-[zx][vu]]
										# -zx may be needed
										# to correct checksums from platform loopback
										# or to flip-cancel -zx in script runL
										# -vu displays network traffic

	32-bit machine-code written emulators
	rta1_{ 1 2 4 8}.[ 1 2 4 8 32 64 256 1024 ] start in rta/tgen.smp ./ra	[-[R][Z][s]]
										# -R = resynchronise network trunk
										# -O = FIFO scheduling. Use only with
										#      an abundance of platform core threads
										# -Z = enter single-step on fault interrupt
										# -s = start emulator in single step
